## 📂 Data Storage & Context Sources (RAG)

These files and folders act as the "Legal Memory" of the AI.

### `data/`
This is your source-of-truth directory. It physically holds the raw, original PDFs (like the *Consumer Protection Act, 2019* and *Model Tenancy Act, 2019*). The ingestion script reads these PDFs to build the AI's legal knowledge base.

### `chroma_db/`
This directory acts as the Vector Database. It stores the semantic meaning (embeddings) of the laws so the AI can search for concepts rather than exact words.
*   **`chroma.sqlite3`**
    This is the lightweight meta-database that ChromaDB uses to keep track of collections, metadata, and indexing locations. It connects the raw vectors to their human-readable textbook counterparts.
*   **`c7949f56-7191-4fd0-92b3-c0affeaf8d6c/` (or similar UUID folders)**
    Whenever you embed documents into ChromaDB, it generates a unique UUID folder (like `c7949...`). Inside this folder, ChromaDB stores the actual heavy binary graphs (HNSW indexes). These binaries represent mathematical coordinates for every sentence in your legal acts, allowing lightning-fast semantic similarity searches.

### `bm25_index.pkl`
While ChromaDB handles *meaning*, this `.pkl` (Pickle) file handles *exact matches*. It is a serialized Python object containing word frequencies. When a user asks about "Section 91", the AI checks this file to find the exact legal code with perfect pinpoint accuracy.

## 🧠 Python Logic & Pipeline Scripts

These scripts are the "Brain" of the AI, executing the data logic.

### `ingest.py`
The "Teacher". It reads the raw PDFs from the `data/` folder, actively chunks them by Chapters and Sections using LangChain, converts them into numbers using `all-MiniLM-L6-v2`, and continuously saves them straight into `chroma_db` and `bm25_index.pkl`.

### `hybrid_search.py`
The "Librarian". Its only job is to query both `chroma_db` and the `bm25_index.pkl` simultaneously, mathematically combine the scores using Reciprocal Rank Fusion (RRF), and hand the best 3 legal paragraphs back to the pipeline.

### `ner_pipeline.py`
The "Extractor". It uses Natural Language Processing (`spaCy` / `en_core_web_sm`) to scan messy human complaints and surgically extract structural data—specifically, who the Petitioner is, who the Respondent is, and exactly how much Money is involved.

### `end_to_end_pipeline.py`
The "Master Orchestrator". This script imports all the above files. It takes the text from `hybrid_search`, the entities from `ner_pipeline`, and streams them to the OpenAI API model under strict prompt restrictions. Rather than spitting out raw text, it takes the LLM's formatted answer, forces it to an 8th-grade reading level, catches AI hallucinations, and builds the final PDF.

## 📄 Templates & Generation Outputs

These handle the physical document output creation.

### `templates/`
A layout directory containing structural blueprints. 
*   **`notice_template.txt`**
    A Jinja2 template file containing placeholders like `{{ petitioner_name }}` or `{{ provision }}`. This acts as the visual backbone of the Legal Demand Notice. The AI injects its answers directly into these brackets.

### `generated_pdfs/`
A sorting folder intended to hold batches of generated output records from the LLM, keeping your root directory clean.

### `Notice_[TIME].pdf` (e.g., `Notice_013426.pdf`)
The actual physical manifestation of the system! These are the Legal Demand Notices successfully exported by the `ReportLab` package inside the `end_to_end_pipeline.py` script.

## ⚙️ Environment & Utilities

These support the Python environment silently.

### `venv/`
The Python Virtual Environment. This massive folder isolates all your installed packages (LangChain, spaCy, FastAPI, ReportLab) perfectly. This ensures your project won't break if your global Windows Python updates its packages.

### `.env`
The hidden Environment Variables file. It securely holds sensitive credentials preventing them from being leaked (like your `OPENAI_API_KEY` or PostgreSQL `DATABASE_URL`). 

### `__pycache__/`
A hidden folder automatically generated by Python. Whenever you run a script, Python compiles it down into bytecode (`.pyc` files) and saves it here so the scripts load drastically faster on the next run. 

### `test.py` & `test_api.py`
Ad-hoc experimentation files. Used for rapidly sketching logic or debugging small functions in isolation before dropping them into the massive `end_to_end_pipeline.py` nervous system.
